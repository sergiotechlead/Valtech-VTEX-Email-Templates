## Overview

This tooling solves a core VTEX development problem: **transactional email templates are edited directly in the VTEX admin**, which has no local preview capability. This server lets you:

- Edit `.hbs` templates locally with hot reload
- Preview rendered emails in a browser at any viewport width
- Edit sample JSON data inline and see changes instantly
- Export compiled HTML ready for upload to VTEX

The templates are written in Handlebars and use the same variables and helpers as VTEX's email rendering engine, so what you see locally matches production output.

Each template is a standalone `.hbs` file that includes its own `<head>`, inline CSS, and all layout markup — there is no shared email layout wrapper. This matches the way VTEX stores templates (one self-contained HTML document per event).

### Template categories

- **Purchase journey (01–13):** Customer-facing transactional emails triggered automatically by VTEX order events.
- **Custom invoice (14):** Triggered by the `backend-core-invoice-generator` app after a PDF is generated and uploaded to S3. The `pdf_url` field in the payload holds the public S3 link.
- **Admin / reports (15–17, 21–23):** Internal VTEX notifications sent to store administrators — order exports, payment reports, catalog exports.
- **Payment & subscription (18–20):** Triggered by payment state changes or subscription billing failures.
- **Account & identity (24–25):** VTEX ID notifications for sign-in alerts and password configuration.

### Email design conventions

- **Max width:** 640 px
- **Font stack:** `Arial, Helvetica, sans-serif`
- **Brand palette:** `#000000` black / `#ffffff` white / `#f9f9f9` light grey
- **Header accent:** 7 px solid black top bar
- **Footer:** Black background with grey legal copy
- **Responsive:** `@media (max-width: 640px)` breakpoint collapses columns
- **MSO compatibility:** VML conditional comments included for Outlook rendering

> To change any visual property, edit the `<style>` block at the top of the `.hbs` file. See [Style System](#style-system) for details.

## Style System

Every template contains a single `<style>` block (≈ lines 18–143) that is the **sole source of truth** for all visual properties. To change a color, size, or spacing value across the email, edit that block — nowhere else.

### Design tokens (documented inside each `<style>` block)

| Token | Value | Used for |
|-------|-------|----------|
| Brand Black | `#000000` | Buttons, borders, `<strong>`, section headings |
| Text Dark | `#1d1d1d` | Greeting `h1`, body links |
| Text Body | `#333333` | Paragraphs, table cells |
| Text Muted | `#9b9b9b` | Footer links, legal notices |
| Background | `#f9f9f9` | Page background, info panels |
| White | `#ffffff` | Email container |
| Border Light | `#eeeeee` | Table row dividers |
| Border Dark | `#333333` | Footer separator |

### CSS classes (no inline style needed)

Structural patterns use named classes so HTML stays clean:

| Class | Element | Purpose |
|-------|---------|---------|
| `.info-box` | `<table>` | Shipping address, payment, delivery panels (grey bg + `margin-bottom:28px`) |
| `.info-box td` | `<td>` | Inner padding `16px 20px` |
| `.info-box p` | `<p>` | Body text inside info panels |
| `.product-table` | `<table>` | Ordered items table |
| `.product-table th/td` | headers/cells | Default left-aligned column |
| `.col-qty` | `th` / `td` | Qty column — centered, `padding: *px 6px` |
| `.col-price` | `th` / `td` | Price column — right-aligned, `white-space:nowrap` |
| `.totals-table` | `<table>` | Order totals / subtotals table |
| `.totals-table td.col-price` | `<td>` | Right-aligned price cell |
| `.h2-flush` | `<h2>` | Heading variant with no top margin (used in help section) |
| `.p-muted` | `<p>` | Small grey text — privacy notices, legal copy |

| `.footer-legal` | `<div>` | Footer legal text block |

### Changing styles

**Color or spacing tweak** — Edit the value in the `<style>` block. The class cascade applies everywhere the class is used.

**One-off adjustment on a single element** — Inline `style="margin-bottom:10px;"` for minor overrides (only the differing property, not a full style dump).

**Adding a new component** — Define a class in the `<style>` block, use it in the markup.

### Note on Gmail

Gmail's mobile app strips `<style>` tags from emails. The remaining inline styles in the HTML cover structural layout properties (max-width, padding, background-color on wrapper tables) that Gmail would otherwise reset. Do not remove these.

## Preview Server

### Routes

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/` | Template gallery — shows all templates as cards |
| `GET` | `/preview/:name` | Full preview UI for one template |
| `GET` | `/preview/:name?raw` | Raw rendered HTML (loaded inside the iframe) |
| `POST` | `/render/:name` | Accepts a JSON body, returns `{ html }` — used by the JSON editor |
| `GET` | `/download/:name` | Downloads the rendered HTML as `<name>.html` |
| `GET` | `/sse` | Server-Sent Events endpoint for live reload |

### Live Reload

`chokidar` watches three directories for any change:

- `templates/` — template `.hbs` files and partials
- `data/vtex/` — JSON data files
- `views/` — preview UI views

On any file event the server broadcasts a `reload` message over SSE. The client script (`public/app.js`) listens and calls `location.reload()`.

Partials in `templates/partials/` are re-registered in Handlebars automatically on every change, so partial edits are reflected without a server restart.

### JSON Editor

The preview page includes a split-pane JSON editor on the right side:

- Displays the sample data file for the active template
- **Auto-apply:** edits are sent to `POST /render/:name` after a 600 ms debounce
- **Apply button:** sends immediately
- **Reset button:** restores the original file data
- The panel can be collapsed with the **✕ JSON** button in the topbar

The iframe is automatically resized to match the email content height after every render and after all images load.

### Viewport controls

Three width presets are available in the preview toolbar:

- **Full** — 100% of the preview container
- **600** — 600 px (typical desktop email client)
- **375** — 375 px (iPhone viewport)

## Build System

`npm run build` (`node build.js`) compiles all templates to static HTML files in `dist/`:

```bash
npm run build

# Build a single template:
node build.js 14-order-invoice-custom
```

The build script:
1. Loads all helpers from `helpers/email.js`
2. Registers all partials from `templates/partials/`
3. For each `.hbs` file in `templates/`, reads the template and its matching data file from `data/vtex/`
4. Compiles and writes `dist/<name>.html`

Output files in `dist/` are gitignored and intended for upload to the VTEX admin.

## Handlebars Helpers

All helpers are defined in `helpers/email.js` and registered globally. They mirror VTEX's built-in template helpers so behaviour is identical in local preview and in production.

### Currency

```hbs
{{formatCurrency value}}
{{formatCurrency value locale="fr-FR" currency="EUR"}}
```

VTEX stores prices as integers in the smallest currency unit (cents). This helper divides by 100 and formats with `Intl.NumberFormat`. Default: `fr-FR` / `EUR`.

### Date

```hbs
{{formatDate creationDate}}
{{formatDate creationDate locale="fr-FR" dateStyle="long"}}
```

Parses an ISO 8601 string and formats it with `Intl.DateTimeFormat`. Default output: `22 juin 2026`.

### Comparison (inline)

| Helper | Usage |
|--------|-------|
| `eq` | `{{#if (eq a b)}}` |
| `ne` | `{{#if (ne a b)}}` |
| `gt` | `{{#if (gt a b)}}` |
| `lt` | `{{#if (lt a b)}}` |
| `gte` | `{{#if (gte a b)}}` |
| `lte` | `{{#if (lte a b)}}` |

### Logical (block)

```hbs
{{#and val1 val2}}both truthy{{/and}}
{{#or val1 val2}}at least one truthy{{/or}}
{{#not value}}falsy{{/not}}
```

### Math

```hbs
{{add a b}}        {{subtract a b}}
{{multiply a b}}   {{divide a b}}
{{percent value total}}
```

Block form:

```hbs
{{#math @index '+' 1}}{{/math}}
```

### String

| Helper | Description |
|--------|-------------|
| `uppercase` | ALL CAPS |
| `lowercase` | all lowercase |
| `capitalize` | First letter upper, rest lower |
| `concat a b c` | Joins arguments |
| `replace str search rep` | String replace (all occurrences) |
| `trim` | Strips whitespace |
| `truncate str 50 "…"` | Truncates with suffix |

### Default

```hbs
{{default value "fallback"}}
```

Returns `value` if it is non-null and non-empty, otherwise returns `fallback`.

### Array / Object

| Helper | Description |
|--------|-------------|
| `first arr` | First element |
| `last arr` | Last element |
| `count arr` | Array length |
| `includes arr val` | Boolean membership check |

### VTEX-specific

```hbs
{{orderStatusLabel status}}
```
Maps VTEX internal status codes to French labels (e.g. `"invoiced"` → `"Expédiée"`).

Supported codes: `payment-pending`, `payment-approved`, `ready-for-handling`, `handling`, `invoiced`, `order-completed`, `canceled`, `cancellation-requested`.

```hbs
{{formatAddress address}}
```
Joins `street`, `number`, `complement`, `city`, `postalCode`, `country` into a single comma-separated string.

```hbs
{{maskCard lastDigits}}
```
Returns `"•••• 4242"`.

```hbs
{{#compare a '===' b}}equal{{else}}not equal{{/compare}}
```
Supports operators: `==`, `===`, `!=`, `!==`, `>`, `<`, `>=`, `<=`.

```hbs
{{#group items by="seller"}}...{{/group}}
```
Groups an array by a property value, then iterates over the sub-arrays.

```hbs
{{json someObject}}
```
Renders an object as formatted JSON — useful for debugging template data.

```hbs
{{times 3}}<tr>...</tr>{{/times}}
```
Repeats a block `n` times.

## Data Files

Each template has a matching JSON file in `data/vtex/` that provides realistic sample data for local preview. The schema mirrors the exact payload VTEX sends to its email template engine at runtime.

### Order-related templates (01–13, 18–20)

Key top-level fields in order-related payloads:

| Field | Type | Description |
|-------|------|-------------|
| `orders` | Array | One or more order objects (multi-order groups have > 1) |
| `orders[].orderId` | string | VTEX order ID |
| `orders[].orderGroup` | string | Group ID for split orders |
| `orders[].state` | string | Order status (VTEX internal code) |
| `orders[].items` | Array | Ordered products |
| `orders[].items[].name` | string | Product name |
| `orders[].items[].price` | number | Unit price in cents |
| `orders[].items[].quantity` | number | Quantity |
| `orders[].items[].imageUrl` | string | Product image URL |
| `orders[].totals` | Array | `{ id, name, value }` line items (Items, Discounts, Shipping, Tax) |
| `orders[].value` | number | Order total in cents |
| `orders[].clientProfileData` | object | Customer name, email, phone |
| `orders[].shippingData.address` | object | Delivery address |
| `orders[].shippingData.logisticsInfo` | Array | Selected SLA / carrier info |
| `orders[].paymentData.transactions` | Array | Payment method details |
| `orders[].creationDate` | string | ISO 8601 timestamp |
| `_accountInfo` | object | Store account metadata (name, country, etc.) |
| `totalValue` | number | Total across all orders in cents |
| `orderGroup` | string | Top-level group ID |
| `split` | boolean | Whether this is a split-order notification |

### Custom invoice template (14)

Template `14-order-invoice-custom` is triggered by the `backend-core-invoice-generator` app, not by a standard VTEX order event. Its payload is simpler:

| Field | Type | Description |
|-------|------|-------------|
| `to` | string[] | Recipient email address(es) |
| `orderId` | string | VTEX order ID |
| `pdf_url` | string | Public S3 URL of the generated invoice PDF |
| `locale` | string | BCP 47 locale (e.g. `fr-FR`) |
| `tradingName` | string | Store trading name |
| `companyName` | string | Company name printed on invoice |
| `_accountInfo` | object | Store account metadata |

### Admin / report templates (15–17, 21–23)

These templates are sent to store administrators by VTEX internal services. Their payloads vary per report type and are not publicly documented by VTEX. The sample data files in `data/vtex/` reflect the actual structure observed in production.

## Deploying to VTEX

1. Run `npm run build` to compile all templates to `dist/`.
2. In the VTEX admin, navigate to **Store Settings → Email templates**.
3. Open the target template (e.g. `new-order`) and paste the contents of the corresponding `dist/` file into the HTML body field.
4. Save and use VTEX's built-in "Send test" feature to verify the output.

> **Note:** VTEX uses its own Handlebars runtime. The helpers in `helpers/email.js` mirror VTEX's built-ins but the compiled HTML written to `dist/` is already fully rendered — no Handlebars syntax reaches the VTEX editor when you use the build output.
